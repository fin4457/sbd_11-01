### Задание 2. Транзакции

2.1. Пользователь пополняет баланс счёта телефона, распишите пошагово, какие действия должны произойти для того, чтобы 
транзакция завершилась успешно. Ориентируйтесь на шесть действий.

**Ответ**:
1. Инициация и проверка: Пользователь вводит номер телефона, сумму пополнения и отправляет запрос; система проверяет корректность введенного номера в базе данных оператора.
2. Блокировка суммы: Банк проверяет наличие нужной суммы на карте пользователя и временно замораживает эти деньги для транзакции.
3. Списание средств: Банк подтверждает успешность авторизации платежа и уменьшает баланс банковского счета пользователя на указанную сумму.
4. Передача данных: Информация об успешной оплате передается через шлюз платежной системы на сторону биллинговой системы мобильного оператора.
5. Зачисление на баланс: База данных мобильного оператора обновляет запись для текущего номера телефона, увеличивая баланс на сумму платежа.
6. Фиксация транзакции: Обе базы данных (банка и оператора) окончательно сохраняют изменения, транзакция закрывается, а пользователь получает уведомление об успехе.


2.1.* Какие действия должны произойти, если пополнение счёта телефона происходило бы через автоплатёж?

**Ответ**:
Если пополнение счета происходит через автоплатеж, ключевое отличие заключается в отсутствии ручной инициации со стороны пользователя. Система автоматизирует первый шаг за счет фоновых процессов (скриптов и планировщиков задач):
1. Триггер события: Специальный робот (планировщик задач/cron) на стороне банка или оператора регулярно проверяет условия. Это может быть наступление определенной даты (например, 1-е число месяца) или падение баланса телефона ниже критической отметки (например, менее 50 рублей).
2. Автоматическое формирование запроса: Как только триггер срабатывает, биллинговая система сама (без участия человека) создает точно такой же запрос на транзакцию, используя ранее сохраненные и привязанные реквизиты карты (токен платежа).
3. Стандартный цикл транзакции: Все последующие шаги базы данных (проверка денег на карте, списание, зачисление оператором и финальный Commit) проходят по такой же схеме, что и при ручном вводе.
4. Результат: Пользователь получает только итоговое уведомление (SMS или Push) о том, что его счет успешно пополнен в фоновом режиме.

### Задание 3. SQL vs NoSQL

3.1. Напишите пять преимуществ SQL-систем по отношению к NoSQL. 

**Ответ**:
1. Строгая согласованность (ACID). Гарантируют полную надежность транзакций. Данные всегда актуальны и согласованы во всех таблицах.
2. Четкая структура данных. Схемы данных (таблицы и связи) жестко определены. Это исключает появление некорректных или дублирующих записей.
3. Мощный стандартный язык запросов. Язык SQL универсален и стандартизирован. Он позволяет выполнять сложные выборки, группировки и объединения (JOIN) одной командой.
4. Минимизация избыточности (Нормализация). Данные разделяются по связанным таблицам. Это экономит память и исключает необходимость обновлять одну запись в нескольких местах.
5. Огромная экосистема и поддержка. Технологии (MySQL, PostgreSQL) развиваются десятилетиями. Существует множество готовых инструментов, библиотек.

3.1.* Какие, на ваш взгляд, преимущества у NewSQL систем перед SQL и NoSQL.

NewSQL — это класс современных СУБД (например, CockroachDB, Google Spanner, TiDB), которые пытаются объединить лучшие черты обеих технологий:
 - Перед традиционным SQL (Преимущество в масштабируемости): Классические SQL-базы тяжело масштабируются горизонтально (из коробки они рассчитаны на вертикальный рост — покупку более мощного сервера). NewSQL изначально спроектированы как распределенные системы. Их можно легко расширять, просто добавляя новые узлы (ноды) в кластер, как в NoSQL, не теряя при этом преимуществ реляционной модели.
 - Перед NoSQL (Преимущество в надежности транзакций): Большинство NoSQL-решений ради высокой скорости и масштабируемости жертвуют строгой согласованностью данных в пользу «согласованности в конечном счете» (Eventual Consistency).
 - NewSQL возвращает полноценную поддержку распределенных транзакций ACID и привычный язык SQL. Разработчику больше не нужно вручную контролировать целостность данных на уровне кода приложения.
 - NewSQL это «золотая середина» — предоставляет безграничную горизонтальную масштабируемость и отказоустойчивость (как в NoSQL) вместе со строгой финансовой надежностью, табличной структурой и мощью SQL-запросов (как в классических СУБД).

### Задание 4. Кластеры

Необходимо производить большое количество вычислений при работе с огромным количеством данных, под эту задачу 
выделено 1000 машин. 

На основе какого критерия будете выбирать тип СУБД и какая модель *распределённых вычислений* 
здесь справится лучше всего и почему?

**Ответ**:
Тип СУБД нужно выбирать на основе структуры данных и требований к консистентности/доступности (CAP-теорема). 
Лучшей моделью распределённых вычислений здесь будет MapReduce (или её современные реализации вроде Apache Spark).

Критерии выбора СУБД
- Природа данных: Если данные строго структурированы — нужны распределенные SQL-решения (NewSQL). Если данные разнородны (тексты, логи, медиа) — выбираются NoSQL-системы (Column-family для аналитики, Key-Value для быстрого доступа).
- Баланс CAP-теоремы: Для аналитики больших данных важнее высокая доступность и устойчивость к разделению (AP), чем мгновенная согласованность (C).

Модель вычислений: MapReduce / Spark

Почему она справится лучше всего:
- Масштабируемость: Модель изначально создана для эффективного разделения одной большой задачи на тысячи независимых подзадач между узлами.
- Отказоустойчивость: При выходе из строя нескольких машин из 1000, система автоматически переназначит их работу другим узлам без перезапуска всего процесса.
- Локальность данных (Data Locality): Вычисления переносятся туда, где данные уже хранятся физически, что минимизирует дорогой сетевой трафик между 1000 машинами.
